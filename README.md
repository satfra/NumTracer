# NumTracer

NumTracer is a header-only C++20 engine that builds and contracts the tensor networks that
appear in quantum-field-theory loop integrands — Lorentz, Dirac, and SU(N) (colour and
flavour) — and **generates flat, FORM-style C++ kernels** from them. It does the tensor
algebra without a symbolic-algebra runtime and without a FORM dependency: each diagram is
contracted numerically over a compact loop frame and lowered to straight-line real
arithmetic, the same shape a FORM reference kernel has.

It is a *general* tensor-tracing engine — the physics lives in the network you hand it, not
in the engine. The reference fixtures here are functional-Renormalization-Group (fRG) flows
for Yang–Mills and QCD, but nothing in the contraction or codegen is specific to them.

## How it works

Kernels are generated by **numeric tracing** — a build-time generator driven from a small
Mathematica front-end:

1. **Front-end** (`mathematica/`). `NumTrace[net, …]` analyses a tensor network written in
   DSL heads (`ntMetric`, `ntVec`, `ntTransProj`, `ntSUNf`, `ntSUNDeltaAdj`, `ntDress`, …) into an
   `NTKernel`: a list of diagrams, each a scalar coefficient times a contraction, plus the
   loop frame and env-id layout. The SU(N) heads are one N-parameterized family (rank N first),
   so colour SU(N_c) and flavour SU(N_f) coexist in one network; the group-diagonal heads
   `ntSUNDiagFund` / `ntSUNDiagAdj` dress each colour/flavour component differently in a single
   trace. `FromFunKit` imports a FunKit flow into the same DSL. Vertex structure-sums — and
   dressed propagator numerators (`"DressingCollection"`, on by default) — are kept eager (no
   product-of-sums or `2^D` monomial blow-up).
2. **Numeric backend** (`MakeNTKernel … "Backend" -> "Numeric"`). Emits a small C++ generator,
   compiles and runs it. The generator contracts each diagram numerically: the Dirac trace as
   4×4 chiral matrix products (including γ5, no `(2n−1)!!` Wick blow-up), the Lorentz network
   by bounded index elimination (no `2^np` projector-mask blow-up), and the colour factor
   folded to a number. Each diagram collapses to a small polynomial in the frame's scalar
   symbols.
3. **Lowering** (`codegen/`, `expr/`). CSE + Horner lower that polynomial to a flat
   straight-line kernel — `trN(const double* f)` trace functions, a `fill()` that computes the
   frame symbols once per call, and the per-diagram assembly.
4. **Consumer.** The committed kernel header is plain C++; it includes only
   `codegen/runtime.hpp` (which supplies `numtracer::complex` and
   `numtracer::compute::{powr,pow,sqrt,fma}`), so the runtime build is dependency-free.

The same contraction primitives are also exposed directly in C++ for hand-built traces and
as the numeric oracle the backend is validated against.

## Layout

Headers live under `include/numtracer/`; include paths use the full prefix, e.g.
`#include "numtracer/numeric/numeric_contract.hpp"`.

| path | role |
|---|---|
| `core/` | foundations: the `constexpr` complex `Cx` (usable as a template parameter), the `Ax<Id, Dim>` axis language and contraction planner, and shared config knobs |
| `expr/` | the compile-time expression algebra (`Zero`/`One`/`Lit`/`Var`/`Add`/`Mul` with structural pruning) and its CSE → real-arithmetic lowering — the kernel lowering stage |
| `network/` | the Lorentz network value (`NetVal`) and its builders (`vec`/`met`/`proj`/`contract`), plus the generator-side numeric colour and Dirac contractions |
| `numeric/` | the **numeric matrix-product backend**: fold a diagram's Dirac trace by 4×4 spinor matrix products and contract the Lorentz network to one polynomial (`MPoly`), then hand it to the lowering driver |
| `dirac/` | typed-out Euclidean γ matrices (Weyl basis) and a sparsity-aware chiral 2×2-block dense γ-trace |
| `sun/` | typed-out SU(2)/SU(3) colour tables plus an `SUNBuilder<N>` runtime oracle they are checked against |
| `dense/` | `dense::DTensor`, a numeric mirror of the contraction engine — the entry-for-entry oracle and the `"Dense"` codegen target |
| `codegen/` | build-time code emission: lower a polynomial to a real straight-line program, share symbols across diagrams, and print the committed kernel header (`runtime.hpp` is its minimal consumer-side support) |
| `cuda/` | a two-phase CUDA quadrature integrator over a grid of external momenta, consuming `__host__ __device__` generated kernels |

## Build & test

The CMake project root is `numtracer/`, **not** the repo root.

```bash
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build -j4
ctest --test-dir build --output-on-failure
```

Builds use `-O3 -march=native -Wall -Wextra`. Tests and benchmarks build only when NumTracer
is the top-level project (`-DNUMTRACER_BUILD_TESTS=ON/OFF` to override). The committed,
generated kernels under `tests/gen/` keep the C++ build and tests dependency-free; regenerate
them only when a flow or the codegen changes.

Each generated kernel is gated against a FORM, dense, or equivalence oracle over random points
(`flow_zq_num`, `flow_za_num`, `flow_za3_num`, `flow_za4_num`, `flow_zaqbq1`, …), alongside the
engine unit tests (`numeric_contract`, `projector_identity`, `projector_fusion`, `dpoly`,
`finiteT_quark`, …) and the per-flavour / per-component / disconnected flows
(`flow_flavour_split`, `flow_flavour_ingroup`, `flow_gluon_condensate`, `discdirac_num`,
`flow_sigl_dirac`, `flow_glu_quark`, `flow_pion_quark`). Opt-in `-DNUMTRACER_SANITIZE` runs the
suite under ASAN/UBSan.

GPU integration tests (CUDA + GSL) are off by default — see `tests/gpu/README.md`:

```bash
cmake -S . -B build-gpu -DNUMTRACER_GPU_TESTS=ON -DCMAKE_CUDA_HOST_COMPILER=g++-14
cmake --build build-gpu -j --target integrate_za3_gpu integrate_za4_gpu
ctest --test-dir build-gpu -L gpu --output-on-failure
```

If long compile-time contraction chains exhaust GCC's `constexpr` budget
("constexpr evaluation exceeded … operations"), link `NumTracer::constexpr_budget` (below) to
raise `-fconstexpr-ops-limit` / `-fconstexpr-loop-limit`.

## Install & use from other projects

```bash
cmake --install build        # default prefix: ~/.local/share/NumTracer
```

This installs the headers plus a CMake package config:

```cmake
find_package(NumTracer REQUIRED HINTS ~/.local/share/NumTracer)
target_link_libraries(my_target PRIVATE NumTracer::NumTracer)
# add NumTracer::constexpr_budget if long chains exhaust GCC's constexpr limits
```

If `wolframscript` is found at configure time, the Mathematica front-end is also installed to
`$UserBaseDirectory/Applications/NumTracer`, so `Needs["NumTracer`"]` resolves from anywhere
(disable with `-DNUMTRACER_INSTALL_MATHEMATICA=OFF`).

## Generating kernels

The codegen runs under `wolframscript` and reuses FunKit's COEN emitter for the scalar
(dressing/boilerplate) half of a kernel — NumTracer owns only the tensor half. The two entry
points:

- `NumTrace[net, "Frame" -> frame, "Args" -> {…}]` — analyse a DSL network into an `NTKernel`.
- `MakeNTKernel[ntk, genFile, kernelFile, tracesFile, "Backend" -> "Numeric", …]` — emit,
  compile, and run the numeric generator, producing the committed kernel. The one-file form
  `MakeNTKernel[ntk, file, …]` defaults to `"Backend" -> "Numeric"`'s cross-check oracle,
  `"Dense"` (the brute-force `dense::DTensor` mirror).

Generated kernels are **self-contained by default**: they include `codegen/runtime.hpp`, are
wrapped in a neutral namespace, and take generic dressing parameters. A consumer with its own
support API redirects the codegen with the `"RuntimeInclude"` / `"SupportNamespace"` /
`"KernelNamespace"` / `"DressingType"` / `"ExtraIncludes"` options.

> The earlier symbolic backends — the expression-tensor `"ET"` path and the invariant-basis
> `"Generate"` path — have been retired; both now error and point at `"Numeric"`.

## Performance

NumTracer trades a small amount of **runtime** for a large amount of **generation time**.

- **Generation.** Because the contraction is plain compile-time C++ with no FORM/`tform`
  trace, generating a flow is roughly **80–175× faster** than the FORM path on the
  quark–gluon vertex flows measured (~18 s per structure vs ~25–56 min); the gap widens on the
  heaviest traces, where the symbolic FORM trace becomes the bottleneck. See `PERFORMANCE.md`.
- **Runtime.** The generated kernels run at roughly **1.0–1.6× FORM** per evaluation (faster
  than FORM on some flows). Absolute ns are machine- and dressing-dependent — the shared
  transcendental dressings dominate and are common to every path; the ratios are the point.
  The lowered kernel is straight-line real arithmetic with no integer division, so it is
  GPU-friendly.

## Documentation

A Sphinx + Doxygen site (narrative guide + C++ API) lives in `documentation/`. It needs a
system `doxygen` plus a Python 3.9–3.12 environment (doxysphinx does not support 3.13+):

```bash
python3.12 -m venv documentation/.venv-docs
source documentation/.venv-docs/bin/activate
pip install -r documentation/requirements.txt
documentation/build.sh                       # or: cmake -B build -DNumTracer_DOCUMENTATION=ON
                                             #     cmake --build build --target documentation
```

The site is written to `documentation/_build/html/index.html`.
